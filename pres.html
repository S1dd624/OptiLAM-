Laminated Additive Manufacturing

LAM builds 3D objects by stacking thin 2D cross-sections of material. A slicer produces N layers as SVG files. Each layer is laser-marked or laser-cut on a flat sheet, then sheets are stacked with adhesive to form the final part.

The Nesting Problem

With 30 slices per object, each must be placed on a 400x400 mm manufacturing bed. Efficient packing minimises wasted material and reduces total machine time.

Three Machine Parameters

Parameter Default Description
Slicing thickness 0.1 mm Z-axis increment per layer
Laser time delay 40 s Dwell time for laser marking
Bed height 78 / 95 mm Initial contact / clearance height, incremented per layer

These three values are input via a Tkinter GUI dialog. Currently, only laser delay is treated as dynamic; Z and time increment progressively in code.

Bounding-Box Nesting vs. Genetic Algorithm

Bounding-Box Approach

  • Each part is enclosed in its axis-aligned bounding rectangle
  • Packing uses shelf or guillotine algorithms on these rectangles
  • Ignores actual part geometry — large waste for irregular shapes
  • Rotation typically limited to 0 and 90 degrees
  • Deterministic: produces one solution
  • Fast, but quality is poor for non-rectangular parts

Genetic Algorithm Approach

  • Works on rasterised pixel geometry (BMPs), not bounding boxes
  • Tests 8 rotation angles per part (45-degree steps)
  • Detects real overlap via bitwise pixel operations
  • Population-based search: many candidates evaluated per generation
  • Evolves over thousands of generations — progressively better
  • Can discover multiple distinct good solutions
The GA trades compute time for solution quality. A 12-hour run can evaluate over 1,000,000 candidate placements — impossible with brute-force enumeration on 30 parts with 8 rotation choices each.

Step 1: SVG Rotator

File: svg_rotator.py / SVGtoBMP- editable.txt. Language: Python. Libraries: cairosvg, Pillow, lxml.

Process

  1. Read each input SVG (named LLL.svg, e.g. 001.svg)
  2. Normalise the canvas: pad viewBox to a square, centre the geometry
  3. Convert all SVG primitives to <path> elements (polygon, rect, circle, ellipse, line, polyline)
  4. For each rotation angle (0, 45, 90, ..., 315 degrees):
    • Apply rotate(angle, cx, cy) transform to all paths
    • Force fill:black on all shapes
    • Render to PNG via CairoSVG, convert to grayscale, threshold to black-and-white, save as BMP

Naming Convention

{LLL}_00_{angle:03d}.bmp
Example: 001_00_045.bmp = part 1, rotated 45 degrees

Why convert to BMP?

SVGs use arbitrary coordinate systems and are fragile for Boolean overlap detection. BMPs are discrete uniform grids. Overlap detection becomes simple pixel arithmetic: BitwiseOR, CountNonZero.

Step 2: Fitness Evaluator

File: Fittnessfn- editable.txt. Language: C#. Framework: .NET. Dependency: OpenCvSharp. Compiled to standalone .exe via Visual Studio.

Chromosome Encoding

Input (args[0]): "X1,Y1,R1,X2,Y2,R2,...,XN,YN,RN"
Three integers per part: X position, Y position, Rotation index

Evaluation Logic

Canvas = blank 100x100 pixel grid (4 mm/pixel = 400x400 mm bed)

For each part i:
    Load BMP file: {i+1:03d}_00_{Ri:03d}.bmp
    Compute kerf scale: scale = 1 + kerf / min(rows, cols)
    Place BMP at (Xi, Yi) on canvas via BitwiseOR

overlapS = sum(individual_areas) - CountNonZero(canvas)

If overlapS > 0:   fitness = -overlapS          (penalty)
If overlapS <= 0:   fitness = left_margin + right_margin   (reward)

Key Design Points

Aspect Detail
Kerf handling Logical dilation ensures physical clearance (10 mm default)
Overlap detection BitwiseOR means overlapping pixels are counted once; difference from sum reveals hidden overlap
Positive fitness Measures unused horizontal margins — tighter packing = higher score
Exit contract Console.WriteLine(fitness) — HeuristicLab reads this single number from stdout

Step 3: Genetic Algorithm (HeuristicLab 3.3)

File: GAcodeforHL- editable.hl. This is a binary project file, editable only inside HeuristicLab on Windows.

How It Works

  1. Initialise a population of random chromosomes (integer vectors of length 3N)
  2. For each chromosome, call the compiled C# fitness evaluator as an external process
  3. Select the fittest individuals for reproduction
  4. Apply crossover operators to combine parent chromosomes
  5. Apply mutation to introduce random variation
  6. Repeat for many thousands of generations

Operational Considerations

Runtime

HeuristicLab does not run well on ordinary laptops. Expect 12 to 21 hour overnight runs on the lab workstation. The .hl file stores all run state internally. Runs must be saved before closing the application.

Crossover Operators (Configurable)

Operator Description
Rounded Average Crossover Averages parent genes, rounds to integer
Discrete Crossover Randomly selects each gene from one parent
Rounded Blend Alpha-Beta (alpha=0.75) Interpolates between parents within an expanded range

These are tunable hyperparameters. Different crossover strategies explore the solution space differently. There is no single best configuration — experimentation over multiple runs is required.

Step 4: SVG Placer + Inkscape Post-Processing

Files: svg_placer_for_ezcad.py, the_final_placeer.py. Language: Python. Library: lxml.

Two Assembly Approaches

File Method Trade-off
svg_placer_for_ezcad.py Wraps SVG children in nested <g transform="rotate/translate"> groups Preserves SVG structure; may cause EzCad2 misinterpretation
the_final_placeer.py Applies rotation and translation directly to path coordinates via trigonometry; emits flat <path> elements No nested groups; cleaner for downstream software

Both multiply X/Y values by 4 to convert from evaluator grid coordinates to millimetres.

Inkscape Post-Processing (Mandatory)

EzCad2 does not reliably interpret <polygon>, nested <g> groups, or certain SVG constructs. This was established through trial-and-error in July 2025.

  1. Open the assembled SVG in Inkscape
  2. Select All, then Edit > Object to Path
  3. Ungroup all nested groups
  4. Delete the bounding rectangle (the bed outline)
  5. Apply Stroke to Path if strokes are present
  6. Save; the resulting SVG contains only flat <path> elements
On July 2, 2025, the first GA-optimised SVG loaded in EzCad2 displayed nested boundary artifacts (one triangle rendered inside another). Inkscape XML editing resolved this, but the process remains manual.

Step 5: EzCad2 Automation (Single-Button Execution)

File: automation_8_fix.py. Language: Python. Libraries: pywinauto, pyautogui, tkinter, win32gui.

Purpose

This script is the glue that makes the pipeline executable from a single button press. It launches EzCad2, imports the assembled SVG, and then drives the entire per-layer laser marking loop programmatically — no manual interaction required after pressing "Start".

Per-Layer Automation Sequence

For each layer in the SVG:
  1. Select first object in EzCad2 object list
  2. Set X position to zero
  3. Apply hatching (opens Hatch dialog, clicks OK)
  4. Enable hatching checkbox
  5. Set marking to BLACK parameters (750 mm/s, 50% power)
  6. Fire Mark(F2) — wait for completion
  7. Disable hatching checkbox
  8. Set marking to BLUE parameters (120 mm/s, 85% power)
  9. Fire Mark(F2) — wait for completion
  10. Delete the current layer from object list
  11. Wait for Next Layer Delay (configurable, default 40s)
  12. Advance to next layer

Key Features

Feature Detail
GUI Tkinter interface: browse SVG file, set next-layer delay, window resize time, marking time, begin-at-layer-N
Pause / Resume Threaded execution with pause/resume buttons; current layer completes before pausing
Retry logic Each sub-step retries up to 3 times on failure; auto-pauses on 3 consecutive failures
Layer counting Parses SVG to count <g> elements; live layer counter in the controls GUI
Error logging All exceptions logged to errorlog.txt with timestamps
Window management Auto-snaps EzCad2 to right half of screen; uses pyautogui for window positioning
This script automates what was previously a fully manual process: opening EzCad2, importing the SVG, manually selecting each layer, setting hatching properties, firing the laser, and deleting the layer. It uses pywinauto to send keyboard shortcuts and click UI elements by their Win32 control identifiers.

Step 6: CNC / G-Code Generator

File: cnc_gen.py. Language: Python. Output: .nc file.

Generated G-Code Structure

%00001
N10 G21                      (metric mode)
Z85                          (initial bed height)

Per layer i:
  N{i}20 G91                 (incremental mode)
  N{i}30 A{delta_x} B{delta_x}   (feed paper by X delta)
  N{i}40 G90                 (absolute mode)
  N{i}50 Y{y - 200 - 9.0}   (position Y, origin-corrected)
  N{i}60 C{rotation}         (orient C-axis)
  N{i}70 Z{78 + i * 0.1}    (lower bed to contact)
  N{i}80 G04 X40             (dwell 40 seconds for laser)
  N{i}90 Z{95 + i * 0.1}    (raise bed for clearance)

Coordinate Transform

y_machine = y_svg - 200 - y_corr

The laser origin is at the bed centreline; SVG origin is top-left. The -200 offset accounts for half the 400 mm bed width. y_corr = 9.0 is an empirical correction factor.

Progressive Increments

Variable Initial Per-layer increment
zHeight (contact) 78 mm +0.1 mm
z1Height (clearance) 95 mm +0.1 mm

Observations

  • The average quality remains negative (many individuals still overlap), but the best solution converges to a high-quality overlap-free placement
  • Multiple runs (e.g. Run 70, Run 71) can produce different valid solutions
  • There is no guarantee of global optimality; longer runs and larger populations improve results
  • The crossover configuration (Rounded Average, Discrete, Blend Alpha-Beta with alpha=0.75) was found through iterative experimentation

Improvement Roadmap

Area Current State Proposed Improvement
Laser software EzCad2 (requires manual Inkscape cleanup) LaserGRBL: open-source, better SVG handling, eliminates the Inkscape step
Evaluation speed Sequential C# evaluator called ~1M times MPI: distribute evaluations across cluster nodes. CUDA: GPU-parallel bitmap operations (BitwiseOR, CountNonZero)
GA hyperparameters Fixed crossover type and rates Systematic tuning of crossover type, mutation rate, population size, and selection pressure across multiple runs
Fitness function 1D (horizontal compactness only) Multi-objective: vertical compactness, material utilisation percentage, cut-path length, thermal distribution
Code structure 6+ svg_placer variants, hardcoded chromosomes Unified configurable script with CLI/config input, unit tests, Docker containerisation
Slicer IceSL (incompatible with lab PC) PrusaSlicer or a custom slicer producing SVG layers in the expected format
SVG post-processing Manual Inkscape Object-to-Path Automate via Inkscape command-line interface or bake transforms into coordinates at Step 4 (the_final_placeer.py approach)
The most impactful near-term improvement is replacing EzCad2 with LaserGRBL, followed by parallelising the fitness evaluator. The GA tuning and fitness function improvements are longer-term research directions.
